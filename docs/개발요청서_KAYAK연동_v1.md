## 0. 한 줄 요약
지금은 **가짜(목) 데이터**로 동작하는 메타서치 앱을, KAYAK Affiliate API에 연결해 **진짜 호텔·가격을 보여주고**, **호텔 카드의 [이동→]을 누르면 실제 예약 사이트로 연결**하며, 그 클릭/예약을 **회원별로 추적해 캐시백을 정산**할 수 있게 만든다.

## 1. 지금 상태 → 목표
| 구분 | 지금(As-Is) | 목표(To-Be) |
|---|---|---|
| 데이터 | 앱 안의 가짜 데이터(MSW 목) | KAYAK 실시간 호텔·가격 |
| 검색/자동완성 | 가짜 목록 | KAYAK 자동완성·검색 결과 |
| [이동→] | 이동 시뮬레이션(가짜 링크) | **실제 공급사 예약 페이지로 새 탭 이동** |
| 캐시백 내역 | 가짜 5건 | KAYAK 리포팅 기반 회원별 실제 내역 |

## 2. 가장 중요한 구조 결정 — "중간다리(중계 서버)"가 반드시 필요
앱(브라우저)이 KAYAK을 **직접 부를 수 없습니다.** 이유 3가지:
1. **정해진 IP에서만 허용**: KAYAK은 화이트리스트한 IP(`58.75.223.130`, 회사 개발실 공용)에서 온 호출만 받음. 손님 브라우저는 IP가 제각각이라 거부됨.
2. **API 키는 비밀**: 키를 앱(브라우저)에 넣으면 누구나 볼 수 있어 유출됨.
3. **브라우저 직접 호출 차단(CORS)**: 보안 정책상 막힘.

→ 그래서 **앱 ↔ 우리 중간다리(중계 서버) ↔ KAYAK** 구조로 만든다. 중간다리가 키를 보관하고, 정해진 IP에서 KAYAK을 대신 호출한 뒤, 결과를 앱에 전달한다.

| 환경 | 중간다리를 어디서 돌리나 | 비고 |
|---|---|---|
| **개발/테스트(지금)** | **회사 개발실 망(58.75.223.130) 안 컴퓨터** | 이미 IP 허용됨 → 바로 가능 |
| **운영(실서비스)** | 고정 IP를 가진 서버 | 그 서버 IP를 KAYAK에 추가 화이트리스트 요청. (클라우드플레어로 하려면 KAYAK이 클라우드플레어 IP대역 허용해야 함 → KAYAK 확인 필요) |

> 예외: **[이동→] 딥링크 클릭은 중간다리 불필요.** 그건 그냥 공개 URL로 새 탭을 여는 것이라 브라우저에서 바로 됨.

## 4. 연동 범위 — 화면 ↔ KAYAK API 매핑
| 앱 화면 | 하는 일 | 연결할 KAYAK API |
|---|---|---|
| 검색(자동완성) | 목적지·호텔 입력 시 후보 | **Autocomplete** `/autocomplete/v1/hotels` |
| 검색 결과(목록) | 목적지의 호텔·가격 목록 | **Hotel Search(다중)** `/api/3.0/hotels` |
| 숙소 상세 | 한 호텔의 공급사별 가격·정보 | **Hotel Search(단일)** `/api/3.0/hotel` |
| [이동→] 아웃링크 | 실제 예약 사이트로 이동 | 검색 응답의 **`bookUri`(딥링크)** + `p=`(회원 라벨) |
| 마이 > 캐시백 내역 | 회원별 예약·캐시백 내역 | **Reporting** `/transactions/hotels` (+`labels` 필터) |
| (보조) 숙소형태·편의시설 한글 이름 | 필터 라벨 | **Static Feed** `/constants-mapping` (숫자ID→이름표) |

> 회원 로그인 자체는 KAYAK이 아니라 **올마이투어 자체 기능**(별도). KAYAK엔 "이 클릭은 어느 회원" 이라는 **라벨(`p=`)** 만 넘긴다.

## 5. 단계(Phase) — 권장 진행 순서
| 단계 | 내용 | 완료 기준 |
|---|---|---|
| P1 연결 테스트 | 중간다리에서 자동완성·검색 1회 호출 → 진짜 응답 수신 | 개발실 IP에서 200 + 실데이터 JSON |
| P2 실데이터 화면 | 자동완성·결과·상세를 실응답으로 교체(필드 매핑) | 3색 데모처럼 화면이 실데이터로 채워짐 |
| P3 딥링크 이동 | [이동→] 클릭 시 `bookUri`에 회원 라벨(`p=`) 넣어 새 탭 | 실제 예약 페이지 도착 + 라벨 전달 |
| P4 캐시백 리포팅 | 테스트 예약 후 Reporting으로 회원별 내역 조회·표시 | 마이>캐시백에 실제 내역(상태/금액) |
| P5 운영 전환 | 운영 호스트·운영 IP·통화/번역 마감 | 손님 대상 오픈 |

## 6. 확정된 결정
- 구조: **중간다리(중계 서버) 필수**. 개발은 회사 개발실 IP에서 구동.
- 캐시백 흐름: KAYAK→올마이투어→유저(리포트 기반). "지급완료"는 **Payment Month 정산완료 + Active** 기준(상세 §개발가이드).
- 딥링크 회원 추적: 반환된 `bookUri`의 **`p=` 자리에 회원 라벨 주입**.
- 앱은 [이동→]만 하고 **예약·결제는 공급사 책임**(중개자 면책 고지 유지).

## 7. 닥스에서 채워야 할 미정 항목 (P5 전까지)
1. **운영(실서비스) 호스트 주소** — RAML 예시는 샌드박스(`sandbox-en-us.kayakaffiliates.com`)만. 운영 도메인 + 한국 마켓 값은 Getting Started 문서 확인.
2. **`p=` 라벨 정확한 표기/인코딩 규칙** — Getting Started > Tracking.
3. **호출 한도(쿼터)** — 너무 잦은 호출 차단 기준.
4. 통화 다국어(엔화 등) — KAYAK 개발팀 회신 대기(현재 KRW 고정).

## 8. 수용 기준 (AC)
- [ ] AC1. 개발실 IP에서 중간다리 경유로 자동완성·검색·상세가 **실데이터** 반환.
- [ ] AC2. API 키가 **앱(브라우저)에 노출되지 않음**(중간다리 환경변수에만).
- [ ] AC3. [이동→] 클릭 시 실제 공급사 예약 페이지로 이동 + `p=` 회원 라벨 전달.
- [ ] AC4. 테스트 예약 후(ET 기준 익일) Reporting에 해당 회원 내역이 조회됨.
- [ ] AC5. 숙소형태·편의시설이 숫자가 아닌 **한글 이름**으로 표시(constants-mapping 매핑).
- [ ] AC6. 가격수집이 끝나지 않은 경우(`onlyIfComplete=false` → 200 + `isComplete=false`)도 화면이 깨지지 않고, 완료될 때까지 재호출(폴링)로 갱신. (`onlyIfComplete=true`로 부르면 202 반복 방식 — 둘 중 택1.)
- [ ] AC7. 현행 3색 데모 화면 레이아웃/동작은 그대로(데이터 출처만 교체).

   - 키는 **중계 서버의 환경변수/시크릿**으로만 보관한다. 소스 코드·프론트엔드·깃에 넣지 않는다.
   - 예) 중계 서버 환경변수: `KAYAK_API_KEY`(검색·정적·자동완성 공용), `KAYAK_REPORTING_KEY`(리포팅), `KAYAK_AFFILIATE_ID`(=`kan_318930_594068`·이건 딥링크에 공개되는 값이라 비밀 아님).
2. **이 폴더의 `.gitignore`(동봉)** 가 다음을 제외한다: `_reference/KAYAK_접속정보_및_QA.md`(키), `_reference/_live_search_sample.json`(대용량 실응답). 새 레포로 복사 후에도 이 `.gitignore`를 유지한다.
3. **실행 환경(IP)**: KAYAK API 호출은 **화이트리스트 IP(`58.75.223.130` = 회사 개발실 망)에서만** 200을 반환한다. 다른 IP(집·클라우드)에서 돌리면 거부된다. 운영 시 그 서버의 고정 IP를 KAYAK에 추가 화이트리스트 요청.
4. **검색 호출 헤더 2개 필수**(개발가이드 §2): `User-Agent`(실 브라우저값) + `x-original-client-ip`(최종 사용자 IP). 자동완성·정적·리포팅은 불필요.
5. 키가 필요한 사람에게는 **깃이 아니라 별도 안전 경로**(사내 시크릿 매니저·1회용 보안 메모 등)로 전달한다.
